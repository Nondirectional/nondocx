# 前端协商实施可视化

## 目标

让 Demo 前端完整表达“协商 → 授权 → 分派 → 专家规划 → review → 提交/回滚 → 质量验收”的新后端工作流，成为可观察、可调试的开发验证界面。

## 已确认事实

- 后端已经通过 SSE 输出 `assistant`、`authorization_required`、`dispatch`、`review`、`commit`、`quality`、`blocked`、`rolled_back`、`doc_changed`、`done` 和 `trace`。
- 当前前端仍以旧 `step(analyze/plan/commit)` 卡片为主；新事件大多退化为单行系统消息。
- 前端已有 trace 面板，但没有按一次协商/一次实施聚合的状态模型；授权和取消是临时按钮，且没有匹配样式。
- `GET /api/trace` 返回 JSONL，页面会重放，但没有清晰的批次分组与执行状态恢复。

## 初始需求

1. 用前端状态机替代旧 `step` 卡片模型，显式表示协商、等待授权、实施中、完成、阻断、回滚、质量未通过等状态。
2. 授权卡展示实施目标、唯一“开始实施”动作和失效/已消费状态；实施中展示固定取消动作。
3. 按实施批次展示 `DispatchPlan`、工具组专家、每个专家 trace、review 操作和提交/质量结果。
4. 统一 SSE 实时事件与 JSONL 回放的 reducer/解码器，避免两套状态逻辑。
5. 更新 HTML/CSS，使授权、执行、错误、回滚、质量结果和 trace 在桌面 Demo 中清晰可读。
6. 每个协商轮次/实施批次渲染为一张可展开时间线卡片，顺序为：协商回复 → 授权 → `DispatchPlan` → 专家 trace → review → 提交/回滚 → 质量；SSE 与 JSONL 回放必须共用 reducer。
7. 当前活跃批次默认展开全部专家 trace；历史批次折叠但保留状态摘要与完整原始 Prompt、thinking、工具参数/结果的展开入口。
8. 质量验收未通过显示“已实施，验收未通过”橙/红色终态，并明确文档已刷新；不得与回滚或提交失败混淆。
9. 取消实施不弹二次确认：点击后立即置灰并显示“正在请求回滚”，等待后端终态。

## 初步验收

- [ ] 新用户能从页面区分“还在协商”和“文档已被实施修改”。
- [ ] 授权、取消、阻断、回滚、质量验收都有稳定、可见且不可误操作的 UI。
- [ ] 同一实施批次的分派、专家过程、review、提交、质量结果在同一时间线内。
- [ ] 重载后 JSONL trace 以相同 reducer 恢复可读状态。
- [ ] 不再渲染旧 ANALYZE→PLAN→COMMIT 专用 `step` 卡片。
