# Cross-Platform Thinking Guide

> **目的**：在实现或调试时，主动思考同一份产物被不同消费者读取时的差异，而不是默认“标准合法 = 所有软件表现一致”。

---

## 问题是什么

很多兼容性 bug 不是“文件坏了”，而是：

- 生成结果 **在规范上合法**
- Apache POI **能重新读回来**
- Microsoft Word **可能也能显示**
- 但 WPS、旧版 Office、第三方查看器 **对默认值的推断不同**

对于 nondocx 这种 docx 库，这类问题尤其常见，因为 `.docx` 本质上是 **ZIP + 多个 XML part + relationship**，不同消费者会对“缺省但合法”的结构做不同解释。

---

## 什么时候要启动跨平台思维

出现以下任一信号时，不要只做库内 round-trip 验证：

- [ ] bug 只在某个具体软件里复现（例如 WPS）
- [ ] 解包后发现 XML “看起来都在”，但 UI 不显示
- [ ] 功能依赖节属性、页面布局、编号、样式继承、默认值推断
- [ ] 示例代码刻意追求“最小 OOXML”
- [ ] Word 正常，但 WPS / 旧版 Office / 在线预览器异常

---

## 三层排查法

### 1. OOXML 层：先确认“有没有写进去”

检查：

- part 是否存在（如 `word/header1.xml`）
- `[Content_Types].xml` 是否声明
- `document.xml.rels` 是否有 relationship
- `document.xml` / `sectPr` / `pPr` 是否真的挂上引用

**目标**：先区分“没写入”还是“写入了但消费者不认/不显示”。

### 2. POI 层：确认 POI 到底帮你做了什么

问自己：

- 是 POI 自动补的，还是调用方显式设的？
- POI 会不会创建 part，但不会补某些默认页面属性？
- 这个 API 是“语义完整”，还是“只写最少必需节点”？

**目标**：别把 POI 的“最小序列化”误认为“最佳兼容序列化”。

### 3. 产品/封装层：当前示例或 API 是否过于极简

问自己：

- 这是库语义必须如此，还是示例为了简洁省掉了上下文？
- 是否应只修改 example / fixture，而不是修改核心默认行为？
- 如果把隐式默认值写成显式 XML，是否更利于教学和兼容？

**目标**：优先把修复范围压在最小、最可验证的位置。

## 常见误区

### 误区 1：POI round-trip 通过，就说明没问题

不对。

POI round-trip 只能证明：

- POI 能写
- POI 也能读回自己写的内容

它**不能证明**其他消费者会以相同方式渲染。

### 误区 2：XML 合法，就说明所有软件都该显示一致

不对。

合法只代表“没违反规范”；并不代表所有软件都会做同样的默认值补全。

### 误区 3：一旦遇到兼容性问题，就该改核心 API 默认行为

不一定。

很多时候更合理的修复层级是：

1. 先改 example / fixture
2. 再补结构断言或兼容性说明
3. 只有确认这是普遍契约问题，才改核心默认行为

---

## 兼容性排查清单

### 实现前

- [ ] 这个功能是否依赖消费者对默认值的推断？
- [ ] 是否存在一个“显式写出”会更稳的结构（如 `pgSz`、`pgMar`）？
- [ ] 我是在改核心契约，还是只是在写教学示例？

### 调试时

- [ ] 对比异常样本和正常样本的解包 XML
- [ ] 把差异收敛到最小结构差异，而不是泛泛怀疑整个功能
- [ ] 区分“数据缺失”和“渲染差异”

### 修复后

- [ ] 至少保留一个结构证据，说明真正触发差异的 XML 是什么
- [ ] 在 spec / guide 中记录该消费者差异
- [ ] 若无法自动化 UI 验证，至少自动化最小结构断言

## nondocx 当前经验

### 页眉页脚 + WPS

已知经验：

- `header*.xml` / `footer*.xml` 存在
- `sectPr` 中也已挂上 `headerReference` / `footerReference`
- POI 往返读取正常
- 但如果该节没有显式 `pgSz` / `pgMar`，WPS 可能对页眉页脚显示更敏感

**当前库契约**：首次通过 `Section.header()` / `Section.footer()` 创建默认页眉页脚时，nondocx 会在页面设置缺失时补齐 `pgSz` / `pgMar`（A4 + 四边 1 英寸边距），但不会覆盖用户已经显式设置的页面属性。

**经验规则**：即使 core 已做兼容性补偿，教学示例仍优先显式写入页面大小和页边距，这样开发者能直接看到 `sectPr` 下的完整结构。 

---

**核心原则**：

> 兼容性 bug 的第一步不是“重写功能”，而是先找出“哪个消费者在依赖哪个隐式默认值”。
