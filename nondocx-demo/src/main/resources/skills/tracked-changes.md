# tracked-changes

## 触发场景

用户要处理修订（tracked changes）：查看现有修订、接受/拒绝修订、以修订模式创作新改动。例如：
"文档里有哪些修订"、"接受所有修订"、"拒绝第 3 条"、"把这段以修订形式改掉"。

## 操作顺序

1. 先 `current_document` 取 doc_id；用修订查询工具列出当前修订及其作者/类型/位置。
2. 接受/拒绝按用户指定范围调用对应修订工具；创作修订用修订创作工具。
3. 改动后必要时 `check_quality` 复核；保存由应用层强制执行。

## 边界

- 区分"修订语义"与"普通文本覆盖"：用户要修订就创作修订，不要把修订当普通段落直接静默改写。
- 接受/拒绝是写操作，会标记 dirty 并触发应用层质检+落盘门控。
- 高层属性类修订（如 pPrChange）若工具不支持，如实说明为 raw-only，不伪造。
- 创作修订时标注清晰的作者与意图，便于用户事后审阅修订列表。

## 停止 / 报告条件

- 指定修订处理完成且质检通过后结束；无修订时如实报告"当前无修订"。
- 用户范围不明确时先确认（全部？某条？），不臆测批量接受/拒绝。
- 接受/拒绝完成后简要说明实际生效的条目数量，避免"已处理"这类无定量描述。
