# audit-quality

## 触发场景

用户要求检查文档质量、解释检查出的问题，或在明确授权下修复可修问题。例如：
"检查一下文档有没有问题"、"质量报告里第 2 条什么意思"、"帮我修掉能修的"。

## 操作顺序

1. 先 `current_document` 取 doc_id。
2. 调 `check_quality` 获得问题清单（error/warning 级别、位置、说明）。
3. 向用户解释每条问题的含义与影响，不堆砌原始字段。
4. 仅当用户明确要求修复时，才对"可由现有工具修复"的问题调用写工具，修完必须复检。

## 边界

- 区分 error 与 warning：error 会阻止保存，warning 允许保存但仍需告知用户。
- 无用户授权不自动改版式或内容；只读检查不改 dirty 状态。
- 无法由现有工具修复的问题必须保留并如实说明，不声称已修复不可修项。
- 解释问题时给出位置（段落/表格/单元格索引）与可能影响，不堆砌原始字段名。

## 停止 / 报告条件

- 解释完问题清单后结束；若已授权修复，则在复检通过后据实报告结果。
- 质检全文已挂入应用层归档（不喂 LLM），你只需回紧凑摘要，避免撑爆记忆窗口。
- 复检发现新问题时继续说明，不在仍有 error 时宣称"全部已修复"。
