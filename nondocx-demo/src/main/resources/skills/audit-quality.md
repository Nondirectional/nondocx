# audit-quality

## 触发场景

用户要求确认文档是否已按要求改好、追问复审结论含义，或在明确授权下补做复审指出的缺失。例如：
"改完了吗帮我确认下"、"复审说部分达成，差在哪"、"把没改到位的补上"。

## 背景：意图复审由 review_intent 承担

文档修改完成后的「意图达成度复审」由只读复审 SubAgent `review_intent` 完成，它会对比用户本轮修改请求与文档现状，返回三态结论（达成 / 部分达成 / 未达成）+ 差异说明。本 Skill 的角色不是替它跑复审，而是**在复审之后**向用户解释结论、并按授权补做缺失。

## 操作顺序

1. 先 `current_document` 取 doc_id。
2. 若本轮尚未复审，调用 `review_intent`（传入本次修改意图）获得复审结论。
3. 读取复审结论的差异说明，向用户解释「哪些达成、哪些缺失/偏差」，不堆砌原始标签。
4. 仅当用户明确要求补做缺失时，才对"可由现有工具修复"的部分调用写工具；补改完后再次调用 `review_intent` 复检。

## 边界

- 复审为**软警告**：即使复审判「未达成」也仍会保存文档，不拦截落盘。你的职责是如实转述结论，让用户自行决定是否补改。
- 无用户授权不自动改内容；只读复审与解释不改 dirty 状态。
- 复审指出的「缺失/偏差」中，无法由现有工具修复的必须保留并如实说明，不声称已修复不可修项。

## 停止 / 报告条件

- 解释完复审结论后结束；若已授权补改，则在复检通过后据实报告结果。
- 复审全文已挂入应用层归档（不喂 LLM），你只需回紧凑结论与差异，避免撑爆记忆窗口。
- 复检仍判「部分达成」或「未达成」时继续据实说明，不在仍有缺失时宣称"全部达成"。
